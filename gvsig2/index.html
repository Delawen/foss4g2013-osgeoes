¿Cómo solucionar esto?

Refactorización grande

Refactorización de gvSIG

  • El acceso a datos: el motor de gvSIG
  • Creación de nuevas APIs
  • Separación estricta de API e implementación
  • Capa de compatibilidad para gvSIG Mobile
  • Mucho más costosa de lo esperado

Se ha reescrito entre otras cosas toda la capa de acceso a datos alfanuméricos, vectoriales y raster. Se ha definido una API coherente para todo el acceso a datos de tal forma que los nuevos drivers que se creen se harán usando esta API.

La separación estricta de las APIs (algo así como un contrato que establece un componente con el resto de componentes del sistema) y las implementaciones permiten mejorar o sustituir componentes enteros del sistema sin que ello afecte al sistema en su conjunto.

Además la API nueva se ha creado de tal forma que tanto gvSIG Desktop como gvSIG Mobile la comparten de tal forma que el desarrollo para ambos componentes es muy parecido.

Ventajas de la nueva arquitectura

  • Corrección de errores
  • Versiones más frecuentes
  • Menos impacto
    = menor tiempo de desarrollo
     = menor coste
  • Nuevas aplicaciones

Se han corregido errores durante la refactorización. La alta modularidad y separación de API e Impl permitirán versiones más frecuentes de solo los componentes que hayan cambiado, pudiendo además controlar mucho mejor el impacto de los cambios

Además al reducir el impacto de los cambios se abaratarán los costes de las correcciones y mejoras al reducirse el tiempo de desarrollo de los mismos.

La alta modularidad de gvSIG permite construir mucho más fácilmente nuevas aplicaciones que usan componentes de gvSIG en su código.

  layer = currentLayer()

  emin = 0.0

    if feature.ELEVATION > emax :
      emax = feature.ELEVATION
    if feature.ELEVATION < emin or emin == 0.0:
